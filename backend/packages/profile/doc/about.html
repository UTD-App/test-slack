utd/profile · v1.0.0

Profile Package

The Profile package turns a user's profile into a single, server-driven, extensible view. The backend aggregates public identity, lets other installed packages contribute their own sections, and serves it all from one endpoint, so the app renders a rich profile without hard-coding which features exist.

What it serves

GET /api/users/{id}/profile returns a single JSON payload built from three layers:

  • publicData — viewer-safe core identity (name, UID, avatar, country, bio, gender, levels). No private fields are leaked. Produced by the base UserDataService::publicData().
  • Contributed sections — one entry per enabled feature package, each under its own key, collected via ProfileContributorRegistry::aggregate(). Gifts contributes today; wallet / follow / vip / family / etc. appear automatically as their packages are installed. Core keys stay authoritative — sections use distinct keys.
  • is_me — a boolean telling the app whether the viewer is looking at their own profile, which drives the editable affordances (edit pencil, "Add a bio", Settings tile, visitor preview).

Admin dashboard

When enabled, the package owns the dashboard's "view a user" screen: it registers a rich Filament infolist (ProfileInfolist) into UserProfileInfolistRegistry, so UserResource renders it instead of the base plain schema. The header pulls only light, bounded data (gift levels, top-N supporters, denormalised social counts); heavy per-user lists live in lazy, paginated relation-manager tabs so opening a whale's profile stays fast. When the package is not installed, the base falls back to its own schema automatically.

Flutter side

The app renders the aggregated payload as the rich profile UI (cover banner, avatar frame, identity, badges, feature grid) and exposes contributed sections to each owning package's widgets via UI slots. It also registers the profile.user Stac object source so UTD Studio screens can bind to the signed-in user's profile. The feature is gated by the backend profile package: when an admin disables it, the app auto-disables the feature and the "Me" tab falls back to the base self-profile fallback.

حزمة الملف الشخصي (Profile)

تحوّل حزمة Profile الملف الشخصي للمستخدم إلى عرض واحد موجّه من الخادم وقابل للتوسعة. يجمع الخادم بيانات الهوية العامة، ويسمح للحِزم الأخرى المثبَّتة بالمساهمة بأقسامها الخاصة، ويقدّم كل ذلك من نقطة نهاية واحدة، فيعرض التطبيق ملفًا ثريًا دون ترميز ثابت للميزات الموجودة.

ما الذي تقدّمه

تُرجع GET /api/users/{id}/profile حمولة JSON واحدة مبنية من ثلاث طبقات:

  • publicData — هوية أساسية آمنة للزائر (الاسم، المعرّف، الصورة، الدولة، النبذة، الجنس، المستويات) دون أي حقول خاصة، عبر UserDataService::publicData() في القاعدة.
  • الأقسام المساهَم بها — قسم واحد لكل حزمة ميزة مفعّلة، كلٌّ تحت مفتاحه الخاص، تُجمع عبر ProfileContributorRegistry::aggregate(). تساهم حزمة الهدايا حاليًا، وتظهر حِزم المحفظة / المتابعة / VIP / العائلة تلقائيًا عند تثبيتها. تبقى المفاتيح الأساسية مرجعية لأن لكل قسم مفتاحًا مميزًا.
  • is_me — قيمة منطقية تُعلِم التطبيق إن كان الزائر يشاهد ملفه الخاص، وهي تتحكم بأدوات التحرير (قلم التعديل، "أضف نبذة"، أيقونة الإعدادات، معاينة الزائر).

لوحة الإدارة

عند التفعيل، تمتلك الحزمة شاشة "عرض المستخدم" في اللوحة: تسجّل infolist غنيًا (ProfileInfolist) في UserProfileInfolistRegistry فيعرضه UserResource بدلًا من المخطط الأساسي البسيط. يجلب الرأس بيانات خفيفة ومحدودة فقط (مستويات الهدايا، أبرز الداعمين، عدّادات اجتماعية مخزّنة مسبقًا)، بينما تعيش القوائم الثقيلة في تبويبات مُحمّلة بكسل ومُرقّمة لتبقى سريعة. وإن لم تُثبَّت الحزمة تعود القاعدة إلى مخططها تلقائيًا.

جانب فلاتر

يعرض التطبيق الحمولة المجمّعة كواجهة ملف ثري (غلاف، إطار الصورة، الهوية، الشارات، شبكة الميزات)، ويكشف الأقسام المساهَم بها لكل حزمة عبر فتحات الواجهة (UI slots). كما يسجّل مصدر كائن Stac باسم profile.user لتربط به شاشات UTD Studio. وتُضبط الميزة عبر حزمة profile الخلفية: عند تعطيلها من الإدارة يتعطّل التطبيق تلقائيًا ويعود تبويب "أنا" إلى الملف الاحتياطي الأساسي.
