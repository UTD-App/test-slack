ابنِ باكدج يُصمّم بصريًا
ويُرسَم native على الموبايل.

UTD Stack منصّة SDUI: الباكدج بتاعك بيعرّف بياناته وعناصره مرّة واحدة، فالعميل يصمّم شاشاته بصريًا في UTD Studio، والـ Base App (Flutter) يرسمها native وقت التشغيل — من غير ما تكتب شاشة واحدة بإيدك.

🧩

عرّف، ماترسمش

تصرّح بعناصرك (elements / widgets / actions) في الـ manifest — الـ Studio يبنيها في الـ palette.

🔌

معزول بالكامل

كل كود الباكدج (backend + flutter) جوّاه. الـ Base يفضل صفر كود خاص بيك.

🎨

WYSIWYG

اللي العميل يصمّمه في الـ Studio = اللي يظهر على الجهاز، بنفس عقد الـ Stac.

الصورة الكبيرة

المعمارية: ٣ أطراف، باكدج من جزئين

المنظومة فيها ٣ أطراف بتتكلّم مع بعض، والباكدج بتاعك بيعيش في طرفين منهم:

🗄️

الجزء الـ Backend (Laravel package)

Service Provider + config/utd_manifest.php + routes + migrations + models. بيعرّف العقد للـ Studio وبيخدم الـ data للموبايل.

📱

الجزء الـ Flutter (AppFeature)

class بيرث AppFeature ويساهم بـ parsers / actions / data sources / routes / providers / ترجمات.

المبدأ الذهبي

الـ Base App عام ومايعرفش منطق باكدجك. كل اللي يعرفه: «هات الـ list/object لمفتاح المصدر ده» و«ارسم الـ type ده». باقي المنطق جوّه باكدجك — وده اللي يخلّي شاشاتك قابلة للتصميم في الـ Studio.

الاتصال

قنوات الاتصال الثلاث

التواصل مع الـ Base بيتم عبر ٣ قنوات منفصلة، كل واحدة لها اتجاه ومفتاح:

القناةالاتجاهالمفتاح / Headerالغرض
ManifestStudio ← BaseX-UTD-Secretاكتشاف الباكدجات وعناصرها وقت التصميم (read)
PushStudio → BaseX-Stac-Keyدفع شاشة Stac المصمّمة (deploy)
RuntimeFlutter ↔ BaseSanctum tokenجلب/إرسال الـ data الفعلية وقت التشغيل
مفتاحان مختلفان تمامًا

X-UTD-Secret (قراءة الـ manifest وقت التصميم) غير X-Stac-Key (نشر الشاشات). الاتنين بيتولّدوا من الـ Studio ويُسلَّموا للعميل يحطّهم في إعدادات الـ Base. أما الـ Runtime فبيستخدم توكن المستخدم العادي (Sanctum).

إيه اللي يخص باكدجك في كل قناة؟

  • Manifest: بتسجّل عقدك في config/utd_manifest.php — الـ Base بيجمّعه تلقائيًا ويخدمه على GET /api/utd/manifest.
  • Push: مالكش دخل — الـ Studio بيدفع الـ Stac والـ Base بيخزّنه. باكدجك بس بيوفّر الـ parsers اللي ترسمه.
  • Runtime: بتسجّل routes الـ API بتاعتك (في الـ ServiceProvider) وبتربط البيانات في الـ Flutter عبر StacDataRegistry.

تشريح باكدج

الباكدج مجلد واحد فيه الجزئين (نموذج باكدج الـ Chat):

structuretree

├─ backend/                         # Laravel package
│  ├─ composer.json               # auto-discovery: extra.laravel.providers
│  ├─ config/utd_manifest.php      # ★ عقد الـ Studio (مصدر الحقيقة)
│  ├─ routes/my.php                # runtime API routes

│  └─ src/
│     ├─ MyServiceProvider.php     # boot(): routes + migrations + registerPackage()
│     ├─ Http/Controllers/         # الـ APIs اللي الموبايل ينده عليها
│     └─ Models/
└─ flutter/                        # Flutter package (depends on utd_app)
   └─ lib/
      ├─ my_package.dart           # exports
      └─ src/
         ├─ my_feature.dart        # ★ extends AppFeature
         └─ stac/                  # parsers / actions / data sources
النموذج الذهني

متى تستخدم إيه؟ — الـ 5 لبنات

قبل ما تكتب سطر واحد، اسأل نفسك سؤال واحد عن كل قطعة في شاشتك: «القطعة دي بيانات بتتعرض، ولا فعل بيتنفّذ، ولا شكل بيرسمه باكدجي؟» الإجابة بتحدّد اللبنة. كل المنظومة مبنية على 5 لبنات بس — لو فهمت إمتى تستخدم كل واحدة، تقدر تفكّك أي باكدج.

اللبنةبتجاوب على السؤالاستخدمها لمّا…مثال
Element
binding		 «إيه الحقل اللي العميل يقدر يربطه؟» عندك قيمة مفردة جوّه صف/كيان (نص، رقم، صورة، تاريخ) العميل عايز يحطها في تصميمه. title، price، avatar، unread_count
Object Source
utdObject		 «منين تيجي بيانات الكيان الواحد في الشاشة دي؟» الشاشة عن كيان واحد (مش قائمة): محادثة مفتوحة، منتج، بروفايل، فاتورة. العميل بيلفّ منطقة ويربط بحقولها. chat.conversation، shop.product، user.profile
List Source
utdList		 «منين تيجي عناصر القائمة المتكرّرة؟» عندك تكرار: قائمة محادثات، منتجات، إشعارات، تعليقات. كل صف بنفس القالب ببيانات مختلفة. chat.conversations، shop.products
Action Element
actionType		 «إيه اللي يحصل لمّا المستخدم يضغط؟» فيه تفاعل: فتح شاشة، إرسال، إضافة للسلة، إعجاب، تعليم كمقروء، اتصال. chat.openConversation، shop.addToCart
Widget + Variants
type		 «إيه الشكل اللي باكدجي لازم يرسمه بنفسه؟» المنطق/الشكل بيختلف حسب الحالة أو محتاج رندر خاص مايتعملش بعناصر الـ core. chat.messages (mine/other/image)، chat.sendButton (send/record)
القاعدة الذهبية للتقسيم

لو القطعة بيقدر العميل يبنيها بعناصر الـ core (Container/Text/Image/TextField + binding) → خليها تصميم العميل (Element / Object Source بس). أمّا لو محتاجة منطق (تختار شكل حسب البيانات، تجيب data بنفسها، تتفاعل مع controller) → دي شغل Widget بتاعك. كل ما تدّي للعميل أكتر وتمسك أنت أقل = باكدج أمرن وأقوى.

شجرة القرار

Backend

عقد الـ Manifest

ملف config/utd_manifest.php هو مصدر الحقيقة الوحيد للـ Studio. بيرجّع array بالشكل ده:

config/utd_manifest.phpphp
return [
  'key'     => 'chat',              // مُعرّف الباكدج (namespace للمفاتيح)
  'name'    => 'Chat',
  'icon'    => 'chat_bubble',
  'screens' => ['conversations', 'conversation'], // شاشات الباكدج المنطقية

  'elements'        => [ /* حقول بيانات قابلة للربط (binding) */ ],
  'object_sources'  => [ /* مصادر «كيان واحد» (utdObject) */ ],
  'action_elements' => [ /* أكشنز (palette + drill-down) */ ],
  'widgets'         => [ /* عناصر مخصّصة يرسمها الـ Flutter parser */ ],
];

elements — الحقول القابلة للربط

كل عنصر = حقل بيظهر في منتقي الـ binding داخل الـ Studio. key لازم يطابق المفتاح اللي بترجّعه في الـ data وقت التشغيل (راجع Registry).

elementsphp
'elements' => [
  ['key'=>'name',  'label'=>'اسم المحادثة', 'type'=>'string',    'screen'=>'conversations'],
  ['key'=>'image', 'label'=>'الصورة',      'type'=>'image_url', 'screen'=>'conversations'],
  ['key'=>'unread_count', 'label'=>'غير المقروء', 'type'=>'int', 'screen'=>'conversations'],
  // hidden:true يمنع ظهوره في المنتقي لكن يفضل متاح للأكشنز/المنطق
  ['key'=>'type', 'label'=>'نوع الرسالة', 'type'=>'string', 'screen'=>'conversation', 'hidden'=>true],
],
الحقلالوصف
keyrequiredمفتاح الحقل (= مفتاح الـ data وقت التشغيل).
labelrequiredالاسم اللي يظهر للعميل.
typeoptionalstring · int · bool · datetime · image_url · id — يحدّد شكل الربط.
screenoptionalأي شاشة منطقية ينتمي لها الحقل.
hiddenoptionaltrue = يُخفى من منتقي الـ binding (يفضل متاح للأكشنز).

object_sources — مصادر الكيان الواحد

للبيانات «المفردة» (مش قائمة) زي بيانات الطرف الآخر في محادثة. العميل بيلفّ منطقة بعنصر Scope (= utdObject) ويربط بحقولها.

object_sourcesphp
'object_sources' => [
  [
    'key'   => 'chat.conversation',        // اسم المصدر (package.key)
    'label' => 'المحادثة الحالية',
    'provides' => [
      ['key'=>'peer_name',   'label'=>'اسم الطرف الآخر',  'type'=>'string'],
      ['key'=>'peer_avatar', 'label'=>'صورة الطرف الآخر', 'type'=>'image_url'],
    ],
  ],
],

ليه الـ Object Source أصلًا؟ وإمتى تحتاجه؟

الفكرة الجوهرية: تفصل «منين تيجي البيانات» عن «إزاي تتعرض». انت بتقول للـ Studio «فيه كيان واحد اسمه كذا وبيقدّم الحقول دي»، وسيبه للعميل يصمّم بيها زي ما يحب — صورة فوق، أو اسم جنب أيقونة، أو هيدر كامل. مالكش دخل بالشكل خالص؛ انت بس بتضمن إن البيانات هتوصل.

  • إمتى تستخدمه: الشاشة (أو منطقة منها) بتتكلّم عن كيان واحد معروف وقت الفتح — مش تكرار. لو لقيت نفسك بتقول «بيانات الــ… الحالي» فده Object Source.
  • الفرق عن utdList: الـ List بيكرّر قالب لكل صف؛ الـ Object بيُرسَم مرّة واحدة بقيم الكيان. القائمة = «اعرض كل المحادثات»، الكيان = «اعرض المحادثة اللي اتفتحت».
  • إزاي بيوصل المعرّف: لمّا العميل يضغط صف في قائمة، أكشن context:'item' بيخزّن بيانات الصف (الـ scope)، والشاشة الجديدة بتقرأ منه. فالـ Object Source بيتملأ من السياق المُمرّر — من غير ما الشاشة تعرف الـ id من فين.
  • الفايدة الكبرى: الهيدر، البروفايل، تفاصيل المنتج… كلها بقت تصميم عميل بالكامل بدل ما تبقى widget جامد بتاعك. ده اللي خلّى هيدر المحادثة (SPEC-07) يطلع من الـ monolith ويبقى Image + Text عاديين مربوطين بـ chat.conversation.

chat.conversation → peer_name, peer_avatar, peer_online. العميل يصمّم الهيدر بيها.

🛍️

shop.product → name, price, gallery, rating, in_stock. صفحة تفاصيل المنتج = تصميم حر.

👤

user.profile → name, avatar, bio, followers. شاشة بروفايل يصمّمها العميل.

action_elements — الأكشنز

الأكشنز اللي تظهر في palette الـ Studio. الـ produces المُسمّى بـ namespace (فيه نقطة) = actionType حقيقي بيتربط بـ StacActionParser في الـ Flutter.

action_elementsphp
'action_elements' => [
  // إدخال أولي (text/audio/image) — مش actionType، بس shape للـ palette
  ['key'=>'text_message', 'label'=>'رسالة نصية', 'produces'=>'text', 'default_shape'=>'input', 'screen'=>'conversation'],

  // Drill-down: ضغط صف بالقائمة → فتح شاشة. context:item يمرّر صف البيانات كامل.
  ['key'=>'open_conversation', 'label'=>'فتح المحادثة',
   'produces'=>'chat.openConversation', 'default_shape'=>'list_item',
   'screen'=>'conversations', 'context'=>'item', 'opens'=>'conversation'],
],
context: 'item'

الأكشن اللي عليه context:'item' بياخد بيانات الصف المضغوط كامل (فيها مثلًا user_id) ويمرّرها للشاشة المفتوحة — ده اللي يخلّي الـ widget في الشاشة الجديدة يعرف أي عنصر اتفتح.

ليه الـ Action Element؟ وإمتى تحتاجه؟

الـ binding بيعرض بيانات — لكنه مايعملش حاجة. أي لحظة المستخدم بيضغط فيها وبيحصل تفاعل (فتح شاشة، إرسال، إضافة، إعجاب، اتصال) محتاجة أكشن. وبتعرّفه في الـ manifest عشان يظهر في الـ palette فيقدر العميل يربطه بزرار/صف من غير ما يكتب كود — والـ Flutter بتاعك بيوصّله بالمنطق الفعلي عبر StacActionParser.

الـ produces هو المفتاح اللي بيربط الاتنين: لو فيه نقطة (namespace زي shop.addToCart) فهو actionType حقيقي لازم يقابله parser في الـ Flutter. لو من غير نقطة (زي text) فهو بس shape للـ palette (إدخال) من غير handler.

نمط الأكشنإمتىالمفاتيح المهمّةمثال
Drill-down (فتح تفاصيل)ضغط صف في قائمة يفتح شاشة كيان. context:'item' + opens + default_shape:'list_item' chat.openConversation
Submit (إرسال/حفظ)إرسال رسالة، حفظ فورم، إضافة تعليق. يقرأ قيمة من حقل (field) ويبعتها للـ API بتاعك. chat.sendMessage، core.login
Mutate (تغيير حالة)إعجاب، تعليم كمقروء، إضافة للسلة، متابعة. context:'item' غالبًا (بيعرف على أنهي عنصر). shop.addToCart، feed.toggleLike
Navigate (تنقّل عام)روح لتاب/شاشة تانية من غير سياق عنصر. core.navigate عبر الفرع الافتراضي. «شوف الكل» → شاشة القائمة
الأكشن نقطة فصل المسؤولية

العميل بيقول «اعمل الأكشن ده هنا» في الـ Studio. إزاي الأكشن يتنفّذ (أي API، أي state) ده شغلك جوّه الـ StacActionParser. فتقدر تغيّر التنفيذ من غير ما العميل يلمس تصميمه.

Backend

تعريف الـ Widgets المخصّصة

الـ widgets هي العناصر اللي باكدجك بيرسمها بنفسه (الـ Flutter parser المسجّل بنفس الـ type). كل widget ممكن يقدّم: خصائص، slots، أو variants.

widgets[]php
[
  'key'     => 'messages',
  'label'   => 'الرسائل (Messages List)',
  'type'    => 'chat.messages',   // = نوع عقدة Stac + parser الـ Flutter
  'icon'    => 'forum',
  'screen'  => 'conversation',
  'scope'   => 'item',             // يقرأ المعرّف من سياق العنصر المُمرّر
  'context' => ['conversation_id'], // مفاتيح السياق اللي الشاشة بتمرّرها للـ widget

  // خصائص قابلة للضبط لكل instance (تظهر في لوحة الإعدادات)
  'properties' => [
    ['key'=>'bubbleRadius', 'label'=>'استدارة الفقاعة', 'type'=>'int', 'default'=>14],
  ],

  // variants: العميل يصمّم شكل كل حالة؛ الـ parser يختار حسب variantField + match
  'variantField' => 'is_mine',
  'variants' => [
    ['key'=>'mine',  'label'=>'رسالتي',       'match'=>['is_mine'=>true]],
    ['key'=>'other', 'label'=>'الطرف التاني', 'match'=>['is_mine'=>false]],
    ['key'=>'image', 'label'=>'صورة',        'match'=>['type'=>'image']],
  ],

  // الحقول النسبية القابلة للربط جوّه قوالب الـ variants
  'bindings' => ['text', 'time', 'status', 'sender_avatar'],
],
الحقلالوصف
type reqنوع عقدة الـ Stac. لازم يطابق StacParser.type في الـ Flutter.
providesحقول الـ widget بيوفّرها للربط بالداخل (مثل هيدر مدمج).
propertiesخصائص لكل instance: color · int · bool · enum · string · icon · binding · field_ref.
slotsمناطق تصميم حرّة جوّه الـ widget (يصمّمها العميل، تُمرَّر للـ parser).
variantField + variantsقوالب فرعية: العميل يصمّم كل واحدة، والـ parser يختار لكل عنصر حسب match. تفاصيل ↓
contextمفاتيح سياق الشاشة بتمرّرها (مثل conversation_id) — يقرأها الـ parser.
bindingsالحقول النسبية المتاحة للربط داخل قوالب الـ variants.
Backend

التسجيل والـ APIs

الباكدج بيُكتشَف تلقائيًا عبر composer.json، والـ ServiceProvider.boot() بيسجّل: routes + migrations + الـ manifest.

src/MyServiceProvider.phpphp
public function boot(): void
{
    // 1) runtime API routes (نفس برِفِكس /api اللي الموبايل بيستخدمه)
    Route::middleware('api')->prefix('api')
        ->group(__DIR__ . '/../routes/my.php');

    // 2) migrations (idempotent بالاسم)
    $this->loadMigrationsFrom(__DIR__ . '/../database/migrations');

    // 3) عقد الـ Studio — guarded عشان الباكدج يفضل يشتغل على base من غير الـ infra ده
    if (class_exists(\App\Support\UtdManifest::class)) {
        \App\Support\UtdManifest::registerPackage(
            require __DIR__ . '/../config/utd_manifest.php'
        );
    }
}
composer.jsonjson
{
  "extra": {
    "laravel": { "providers": ["Utd\\Chat\\ChatServiceProvider"] }
  }
}
الـ Runtime APIs

الـ endpoints اللي بتجيب/تبعت الـ data (قوائم، تفاصيل، إرسال، رفع) تنفيذ داخلي عندك — الموبايل بيناديها مباشرة بتوكن المستخدم. الـ Studio مايعرفهاش ومايحتاجهاش.

Flutter

واجهة AppFeature

قلب باكدجك على الموبايل: class بيرث AppFeature ويساهم بكل حاجة. كله اختياري ماعدا id وdisplayName.

lib/src/my_feature.dartdart
class MyFeature extends AppFeature {
  @override String get id => 'com.acme.reactions';
  @override String get displayName => 'Reactions';

  @override
  Future<void> initialize() async {
    registerMyStacSources(_repo);     // اربط الـ data sources
  }

  @override
  List<StacParser> getStacParsers() => const [MyWidgetParser()];

  @override
  List<StacActionParser> getStacActionParsers() => const [MyActionParser()];

  @override
  List<SingleChildWidget> getProviders() => [/* BLoCs/repos */];
}
الميثودالمساهمة
id / displayName reqمُعرّف عكسي للنطاق + اسم معروض.
initialize()تهيئة + تسجيل الـ data sources في StacDataRegistry.
getStacParsers()widgets مخصّصة (type → widget). لازم stateless وتجيب deps من الـ BuildContext.
getStacActionParsers()أكشنز مخصّصة (actionType → handler).
getProviders()BLoCs/repositories على مستوى الـ app.
getRoutes()مسارات native (GoRoute) — استخدم prefix بالـ id.
getUiContributions()عناصر في slots جاهزة (مثل تاب bottom-nav).
getTranslations()ترجمات locale → key → نص (مفاتيح بـ namespace).
getUserDataExtensions() · getRoleDefinitions() · getSettingDefinitions()توسيع بيانات المستخدم/الأدوار/الإعدادات.
Flutter

ربط البيانات — StacDataRegistry

الجسر اللي بيوصّل عناصر الـ Stac المربوطة (utdList / utdObject) ببياناتك الحقيقية. الـ Base يسأل «هات المصدر ده» وانت بترجّع الـ data — من BLoC أو repo أو API.

lib/src/stac/my_sources.dartdart
void registerMyStacSources(MyRepo repo) {
  // قائمة → utdList بمفتاح المصدر ده
  StacDataRegistry.instance.registerList('chat.conversations', () async {
    final rooms = await repo.getRooms();
    return rooms.map((r) => {
      'name': r.name, 'image': r.image,   // المفاتيح = elements في الـ manifest
      'unread_count': r.unread, 'user_id': r.userId,
    }).toList();
  });

  // كيان واحد → utdObject
  StacDataRegistry.instance.registerObject('chat.conversation', () async => {
    'peer_name': scope.name, 'peer_avatar': scope.image,
  });
}
المفاتيح لازم تطابق

المفاتيح اللي بترجّعها هنا (name, image…) لازم تطابق key بتاع الـ elements/provides في الـ manifest — عشان الـ binding يتحل بدون أي mapping إضافي.

لو المصدر مش متسجّل أو رمى استثناء، الـ Base يرجّع قائمة/كائن فاضي — الواجهة تتدهور بهدوء بدل ما تكسر.

Flutter

الـ Binding داخل Stac

الـ Studio بيُخرِج عُقَد فيها binding، والـ Base يحلّها من الـ data وقت التشغيل عبر StacBinding.resolve().

binding (Stac)json
// قبل الحل (من الـ Studio)
{ "type":"text",  "binding":"name" }
{ "type":"image", "binding":"peer_avatar" }

// بعد الحل (اللي الـ renderer يفهمه)
{ "type":"text",  "data":"أحمد" }
{ "type":"image", "src":"https://…", "imageType":"network" }
  • مفاتيح نسبية أو كاملة: name أو chat.conversations.name — بيتاخد آخر segment للبحث.
  • القوائم: utdList له source وitemTemplate بداخله bindings نسبية لكل صف.
  • الكيان الواحد: utdObject له source + child يُرسَم مرّة بقيم المصدر.
  • الإخفاء الشرطي: visibleBinding يشيل العنصر لو القيمة falsy (نقطة أونلاين، عدّاد = 0…).
  • التواريخ: قيمة ISO تتحوّل تلقائيًا لـ HH:mm النهارده أو yyyy/MM/dd.
Flutter

Widgets مخصّصة (StacParser)

لكل widget في الـ manifest، اكتب parser بنفس الـ type. لازم يكون stateless ويجيب الـ dependencies من الـ BuildContext وقت الرسم.

my_widget_parser.dartdart
class MyWidgetParser extends StacParser<Map<String, dynamic>> {
  const MyWidgetParser();

  @override String get type => 'chat.messages';       // = widget.type
  @override Map<String,dynamic> getModel(Map<String,dynamic> j) => j;

  @override
  Widget parse(BuildContext ctx, Map<String,dynamic> model) {
    final variants = (model['variants'] as Map?)?.cast<String,dynamic>();
    // اقرأ context/properties، هات الـ data من BLoC عبر ctx، وارسم.
    return MyListView(variants: variants);
  }
}

لرسم شجرة Stac جوّه parserك (مثل قالب variant بعد حلّ الـ binding):

render a subtreedart
final resolved = StacBinding.resolve(template, rowData);  // حلّ الـ bindings
return Stac.fromJson(resolved, context) ?? const SizedBox.shrink();

Actions مخصّصة (StacActionParser)

لكل produces بـ namespace في الـ manifest، اكتب action parser. الـ Base يمرّر صف القائمة المضغوط تحت item (للأكشنز اللي context:'item').

my_action.dartdart
class OpenConversationAction extends StacActionParser<OpenConvModel> {
  @override String get actionType => 'chat.openConversation';

  @override
  FutureOr onCall(BuildContext ctx, OpenConvModel m) {
    // خزّن سياق العنصر (peer) ثم انتقل للشاشة المصمّمة
    ChatConversationScope.set(userId: m.item['user_id'], name: m.item['name']);
    ctx.go(m.route ?? '/s/conversation');
  }
}
القدرة المميزة

الـ Variants (قوالب فرعية)

أي widget منطقه بيختلف حسب الحالة (رسالتي/الطرف الآخر، صورة/ملف/صوت، إرسال/تسجيل) يقدّم variants: العميل يصمّم شكل كل حالة بصريًا، والـ parser يختار القالب المناسب لكل عنصر وقت التشغيل.

إزاي تختار القالب في الـ parser

variant pickdart
Map<String,dynamic>? pickVariant(msg, variants) {
  if (msg.type == 'image' && variants['image'] != null) return variants['image'];
  return msg.isMine ? variants['mine'] : variants['other'];
}
مثال حقيقي: زرار واحد بيتقلب

زرار الإرسال/التسجيل (chat.sendButton) = widget واحد بـ variants {send, record}. لو مربع الكتابة فاضي يعرض قالب record، وفيه نص يعرض send — نفس آلية الرسائل بالظبط. لو الحالة ملهاش قالب مصمّم، يرجع لأيقونة Material افتراضية.

إمتى تحتاج Variant؟ (الاختبار البسيط)

اسأل: «نفس النوع من العنصر بيتعرض بأكتر من شكل حسب بياناته؟» لو الإجابة آه — ده variant. لو العنصر شكله ثابت دايمًا → مايحتاجش variant، بس binding. الـ variant بيدّي العميل يصمّم كل حالة على حدة، وانت في الـ parser بتختار الحالة المناسبة لكل عنصر وقت التشغيل عبر variantField + match.

المحور اللي بيختلفالـ variantFieldالـ variants
صاحب الرسالة (شات)is_minemine / other
نوع المحتوى (شات)typetext / image / file / audio
حالة الزرار (شات)modesend / record
نوع الإشعارkindlike / comment / follow / system
حالة الطلب (متجر)statuspending / shipped / delivered / cancelled
نوع منشور الـ feedpost_typetext / photo / video / poll
variant مش property

لو الفرق مجرّد قيمة (لون، استدارة، حجم) → خليه property على العنصر، مش variant. الـ variant للحالات اللي التخطيط/التركيب نفسه بيختلف فيها (فقاعة نص مقابل صورة بكابشن مقابل مشغّل صوت بـ waveform). قاعدة سريعة: لو ينفع تعمله بـ binding + property → اعمله كده؛ سيب الـ variants للاختلاف الحقيقي في الشكل.

التفكيك

playbook: تفكيك باكدج جاهز بـ UI ثابت

عندك باكدج شغّال بشاشاته المرسومة بإيدك (Flutter ثابت)؟ تحويله لـ SDUI مش إعادة كتابة — ده إعادة تقسيم. بتفصل اللي ثابت ومنطقي (شغلك) عن اللي المفروض العميل يصمّمه. النتيجة: نفس الفيتشر، بس العميل يقدر يطلّعها بأي شكل من غير ما يلمس كودك.

نقلة التفكير

بدل ما تسأل «إزاي أرسم الشاشة دي؟»، اسأل «إيه البيانات اللي الشاشة دي محتاجاها، وإيه التفاعلات، وإيه الأجزاء اللي شكلها لازم يتغيّر؟» الإجابات هي بالظبط الـ manifest بتاعك.

الوصفة في 6 خطوات

  1. افرز الشاشات لقوايم وكيانات

    كل شاشة فيها تكرار؟ دي utdList + elements. فيها كيان واحد (تفاصيل/بروفايل)؟ دي object_source. اكتب كل مصدر بمفتاح package.key.

  2. استخرج الحقول (Elements)

    بُص لكل قيمة في الـ UI (نص/رقم/صورة/تاريخ) واسأل «العميل هيعرضها؟». لو آه → element بـ key يطابق الـ JSON اللي بترجّعه من الـ API. اللي بيستخدمه المنطق بس (زي user_id) خليه hidden:true.

  3. حدّد التفاعلات (Actions)

    كل ضغطة بتعمل حاجة = action_element. اللي بيفتح تفاصيل عنصر حطّ عليه context:'item' + opens. اللي بيبعت/يغيّر حالة سمّيه بـ namespace (produces:'pkg.doX') عشان تربطه بـ parser.

  4. اعزل اللي شكله بيتغيّر (Variants)

    أي عنصر بيتعرض بأكتر من تخطيط حسب بياناته → widget بـ variantField + variants. سيب كل حالة للعميل يصمّمها، وانت تختار في الـ parser.

  5. سلّم الباقي للعميل

    أي جزء يتبني بـ Container/Text/Image/TextField + binding (هيدر، بطاقة، فورم بسيط) — متعملوش widget. سيبه تصميم عميل فوق Object Source. ده اللي يخلّي الباكدج مرن.

  6. وصّل الـ Flutter

    registerList/Object لكل مصدر (مفاتيح تطابق الـ elements)، StacParser لكل widget بـ variants، StacActionParser لكل actionType. شيل الشاشات المرسومة القديمة.

مثال مُفكَّك: من «شاشة متجر ثابتة» لباكدج SDUI

باكدج متجر قديم فيه شاشتين مرسومتين: قائمة منتجات وتفاصيل منتج. كده بيتفكّك:

في الـ UI الثابتبيتحوّل لـفي الـ manifest
GridView بمنتجاتList Sourceshop.products (utdList)
اسم/سعر/صورة المنتج في الكارتElementsname, price, thumb, discount

قبل/بعد

قبل: 2 شاشة Flutter جامدة، أي تغيير شكل = نسخة جديدة من الباكدج.
بعد: 2 مصدر + 1 widget بـ 3 variants + 2 أكشن. كل عميل يطلّع متجره بشكله، وانت بتصون المنطق بس (الجلب، السلة) في مكان واحد.

ابدأ صغير، فكّك أكتر بعدين

مش لازم تطلّع كل حاجة widget من اليوم الأول. ابدأ بـ utdList + elements (شغّال من غير parser)، وبعدين فكّك العناصر اللي شكلها بيختلف لـ widgets بـ variants لمّا تحتاج. ده بالظبط مسار الشات: monolith ضخم → chat.messages + chat.sendButton + object_source للهيدر + TextField عام للكتابة.

عملي

أول package — خطوة بخطوة

هنبني باكدج بسيط «Announcements» (قائمة إعلانات) من الصفر:

  1. عرّف العقد (backend)

    في config/utd_manifest.php: key=announcements, screen list, وelements: title, body, image, created_at.

  2. سجّل الباكدج (backend)

    في ServiceProvider.boot() نادِ UtdManifest::registerPackage(require .../utd_manifest.php) + سجّل routes الـ API بتاعتك + migrations. وأضف الـ provider في composer.json.

  3. اربط الـ data (flutter)

    في initialize() داخل الـ AppFeature: StacDataRegistry.instance.registerList('announcements.list', () async => repo.getAll()) — بمفاتيح تطابق الـ elements.

  4. سجّل الـ Feature (flutter)

    أضِف MyFeature() لقائمة الـ features في الـ Base App (feature_registry). من غير parsers مخصّصة هتستخدم utdList الجاهز.

  5. صمّم في الـ Studio

    العميل يفتح الـ Studio → يلاقي باكدج «Announcements» في الـ palette → يحط List مصدره announcements.list ويربط title/image → ينشر.

  6. اتأكد على الجهاز

    flutter build apk + تثبيت → الشاشة بتظهر ببياناتك الحقيقية بالشكل اللي العميل صمّمه. لو محتاج عنصر بمنطق خاص، أضِف StacParser.

قائمة تحقّق الباكدج

Backend

  • utd_manifest.php فيه key/screens + elements/widgets/actions
  • المفاتيح تطابق data الـ runtime
  • ServiceProvider يسجّل manifest + routes + migrations
  • auto-discovery في composer.json

Flutter

  • AppFeature مسجّل في الـ Base App
  • registerList/Object لكل مصدر
  • StacParser لكل widget (stateless)
  • StacActionParser لكل actionType
  • flutter analyze نضيف + اختبار على جهاز
المبدأ النهائي

لو الـ Base محتاج يعرف أي حاجة عن منطق باكدجك — يبقى في حاجة مكانها غلط. كل المنطق جوّه الباكدج؛ الـ Base بس بيجمّع، يخدم، ويرسم.
