UTD Studio SDK

UTD Studio is the Flutter Server-Driven UI (SDUI) runtime. It renders screens authored in the UTD Studio editor from JSON delivered by the backend — so a screen can change without an app-store release. The package is a curated glue layer over a vendored fork of the open-source Stac framework (bundled byte-for-byte under vendor/) plus the UTD layer on top.

Consumers import only one barrel — package:utd_studio_sdk/utd_studio_sdk.dart — never package:stac/.... The barrel re-exports a deliberately small slice of the Stac API (Stac, StacParser, StacActionParser, StacFormScope) plus the UTD facade and runtime.

What it adds on top of Stac

  • The UtdStudio.init(StudioConfig) facade — one boot entry point that reproduces the exact Base boot order: Stac.initialize(parsers, actions) once, then screenSource.init() (open the cache), then a fire-and-forget syncAll() in the background — preserving the no-spinner-flash startup.
  • A port-based runtime (StudioRuntime) — a process-wide holder for the injectable seams the host app implements: StacTransport (HTTP), KeyValueCache (Hive/SharedPreferences/memory), and optional AppNavigator, ThemeSource, LocaleSource, ToastSink, UserSession and a translate port. The SDK never imports app code — it only talks to these contracts.
  • Custom UTD parsers — the utd* widget set (utdList, utdObject, utdTabs, utdTextField, utdScroll, utdSized, utdPositionedDirectional, …) layered over the upstream primitives.
  • Generic core.* actions — core.navigate, core.back, core.openDialog, core.closeDialog, core.toggleTheme, core.setLocale — each resolving its dependency from StudioRuntime.
  • localizeStac i18n binding — localises a screen's translatable Text nodes before render. A node is translated when it carries a tKey, a t.* binding, or a data literal that is itself a catalog key. Resolution goes through the app's translate port, so Studio labels follow the app language and the dashboard translation catalog.
  • Screen store (StacScreenStore) — cache-first fetch with version sync (it replaces upstream's StacService; the barrel deliberately does not re-export StacService).
  • Shared registries — the data-binding registry (StacDataRegistry) + template binder (StacBinding), and the StudioSlotRegistry (the Studio analogue of native UI slots: a package injects a card into a screen it does not own).
  • Defensive coercion (StacCoerce) — targeted "force parsing" of known string-typed fields so a mistyped server value (e.g. text.data as an int) coerces to String instead of blanking the whole screen.

How a screen renders

StacDynamicScreen(screenName: 'profile') reads the resolved screen source from StudioRuntime, pulls the cached JSON (no spinner flash), runs the i18n + coercion passes, and hands the tree to the vendored Stac engine. Feature packages contribute parsers / actions / data sources and slot cards through the SPI (StacContributor), and screens bind to data registered via UtdStudio.registerObject / registerList.

Why a vendored fork (not a pub dependency)

The four upstream packages — stac, stac_core, stac_framework, stac_logger — are copied verbatim under vendor/ and consumed via dependency_overrides. This pins exact versions, lets UTD ship parsers/actions against a stable engine, and avoids a deep symbol rename of ~696 vendored files on every re-merge. Internal symbols keep their Stac* names by design; branding is applied at the package + UtdStudio facade level. See UPSTREAM.md.

Footgun: pub honours dependency_overrides only from the entrypoint (app) pubspec. The vendored-stac override block MUST be duplicated in every app that depends on this SDK, or you get two copies of the engine and parser registration silently no-ops. See installation.html.

حزمة UTD Studio

UTD Studio هي بيئة تشغيل الواجهات الموجَّهة من الخادم (SDUI) في فلاتر. تعرض شاشاتٍ مُنشأة في محرّر UTD Studio انطلاقًا من JSON يرسله الخادم، فيمكن تغيير الشاشة دون إصدار جديد على المتجر. الحزمة هي طبقة ربط منسّقة فوق نسخة مُضمَّنة (fork) من إطار Stac مفتوح المصدر (مُضمَّن حرفيًا تحت vendor/) مع طبقة UTD فوقه.

يستورد المستهلكون ملفًا واحدًا فقط — package:utd_studio_sdk/utd_studio_sdk.dart — وليس package:stac/... أبدًا. يعيد هذا الملف تصدير شريحة صغيرة مقصودة من واجهة Stac (Stac وStacParser وStacActionParser وStacFormScope) إضافةً إلى واجهة UTD وبيئة التشغيل.

ما الذي تضيفه فوق Stac

  • واجهة UtdStudio.init(StudioConfig) — نقطة إقلاع واحدة تعيد ترتيب الإقلاع كما في القاعدة: Stac.initialize مرة واحدة، ثم screenSource.init() (فتح الكاش)، ثم syncAll() في الخلفية دون انتظار — مع الحفاظ على إقلاع بلا وميض تحميل.
  • بيئة تشغيل قائمة على المنافذ (StudioRuntime) — حامل على مستوى العملية للمنافذ القابلة للحقن التي يطبّقها التطبيق المضيف: StacTransport (HTTP)، وKeyValueCache (Hive/SharedPreferences/الذاكرة)، إضافةً إلى المنافذ الاختيارية AppNavigator وThemeSource وLocaleSource وToastSink وUserSession ومنفذ translate. لا تستورد الحزمة كود التطبيق إطلاقًا، بل تتحدث عبر هذه العقود فقط.
  • مُحلِّلات UTD المخصّصة — مجموعة عناصر utd* (utdList وutdObject وutdTabs وutdTextField وutdScroll وutdSized وutdPositionedDirectional …) فوق عناصر Stac الأساسية.
  • إجراءات core.* العامة — core.navigate وcore.back وcore.openDialog وcore.closeDialog وcore.toggleTheme وcore.setLocale — يجلب كلٌّ منها اعتماده من StudioRuntime.
  • ربط الترجمة localizeStac — يترجم عقد Text القابلة للترجمة قبل العرض. تُترجم العقدة إذا حملت tKey أو ربطًا يبدأ بـt. أو نصًّا في data هو نفسه مفتاح في الكتالوج. تمر الترجمة عبر منفذ translate في التطبيق، فتتبع تسميات Studio لغة التطبيق وكتالوج ترجمة اللوحة.
  • مخزن الشاشات (StacScreenStore) — جلب يعتمد الكاش أولًا مع مزامنة الإصدارات (يحلّ محل StacService الأصلي؛ ولا يعيد الملف تصدير StacService عمدًا).
  • سجلّات مشتركة — سجلّ ربط البيانات (StacDataRegistry) ورابط القوالب (StacBinding)، وStudioSlotRegistry (نظير فتحات الواجهة الأصلية في Studio: حزمة تحقن بطاقة في شاشة لا تملكها).
  • تحويل دفاعي (StacCoerce) — "فرض تحليل" مُوجّه للحقول المعروفة من نوع نص، فإذا وصلت قيمة بنوع خاطئ من الخادم (مثل text.data كـint) تُحوَّل إلى String بدلًا من إفراغ الشاشة كاملة.

كيف تُعرض الشاشة

يقرأ StacDynamicScreen(screenName: 'profile') مصدر الشاشة المُحدَّد من StudioRuntime، ويسحب JSON المخزَّن (بلا وميض)، ثم يُجري مرحلتي الترجمة والتحويل، ويسلّم الشجرة لمحرّك Stac المُضمَّن. تساهم حِزم الميزات بالمحلّلات والإجراءات ومصادر البيانات وبطاقات الفتحات عبر واجهة المساهمة (StacContributor)، وتربط الشاشات ببيانات مسجَّلة عبر UtdStudio.registerObject / registerList.

لماذا نسخة مُضمَّنة (لا اعتماد من pub)

الحِزم الأربع الأصلية — stac وstac_core وstac_framework وstac_logger — منسوخة حرفيًا تحت vendor/ وتُستهلك عبر dependency_overrides. هذا يثبّت إصداراتٍ دقيقة، ويتيح لـUTD شحن محلّلات/إجراءات فوق محرّك مستقر، ويتجنّب إعادة تسمية عميقة لنحو ٦٩٦ ملفًا مُضمَّنًا عند كل دمج. تحتفظ الرموز الداخلية بأسماء Stac* عمدًا، وتُطبَّق العلامة التجارية على مستوى الحزمة وواجهة UtdStudio. انظر UPSTREAM.md.

مزلق شائع: يحترم pub الكتلة dependency_overrides من pubspec الخاص بـنقطة الدخول (التطبيق) فقط. يجب تكرار كتلة تجاوز stac المُضمَّن في كل تطبيق يعتمد على هذه الحزمة، وإلا حصلت على نسختين من المحرّك ويفشل تسجيل المحلّلات بصمت. انظر installation.html.
