Authentication — Installation

How to add the package to a host app, wire it into the platform, and the base API it expects.

English

1 · Add the dependency

Add a path dependency to the host app's pubspec.yaml (the package lives under packages/authentication):

dependencies:
  authentication:
    path: packages/authentication

Then fetch packages:

flutter pub get

The package itself depends on utd_app (the feature contract) and utd_studio_sdk (the server-driven screen renderer), both as path deps.

2 · Register the feature

Add AuthFeature() to the list of features the platform builds. Auth is a core feature (isCore = true), so register it first:

import 'package:authentication/authentication.dart';

List<AppFeature> buildFeatures() {
  return [
    AuthFeature(),   // core — register first
    // ...other features
  ];
}

That single registration is enough. The platform's FeatureRegistry drives the rest automatically:

HookWhat it wires
initialize()Builds the api service → datasource → repository → use cases, fills AuthLocator (for Studio actions), and creates the BLoCs. Awaited before any route builds.
getRoutes()Returns AuthRoutes.routes() — splash / intro / login / register / add-information / recover-password / on-boarding / privacy.
getProviders()Provides SplashBloc, LoginBloc, RegisterBloc, AddInformationBloc, RecoverPasswordBloc app-wide.
getTranslations()Merges the auth.* EN + AR catalog into the app's translations (app name interpolated).
For server-driven screens, pass the auth Studio action parsers (core.login / core.register / core.forgotPassword / core.completeProfile) into UtdStudio.init via StudioConfig.extraActions. They read use cases from AuthLocator, which initialize() fills. If a screen is unpublished, the native fallback page runs instead.

3 · Base API contract

The package talks to the base host under the /api prefix. Endpoints (from AuthApiService):

Use caseEndpoint
LoginPOST /auth/login
RegisterPOST /auth/register
Add informationPOST /profile/update
Recover — send OTPPOST /auth/forgot-password/send-otp
Recover — verify codePOST /auth/forgot-password/verify-code
Recover — resetPOST /auth/forgot-password/reset-otp

Login request & response

Login posts { "type": "email_pass", "email", "password" }. On success the response data must carry the token under data.auth_token (the package reads json['auth_token'] into the session and saves it):

POST /api/auth/login
{ "type": "email_pass", "email": "user@x.com", "password": "secret" }

200 OK
{
  "message": "Logged in",
  "data": { "id": 12, "is_first": false, "auth_token": "..." }
}

The login response carries no full user object (only id / is_first / auth_token); the package then loads the current user from /my-data into the session.

Bad credentials → 422

On invalid credentials (or validation failures) the base must return HTTP 422 with a message. The package surfaces that message as the error toast — it does not retry. Register similarly returns the new data.auth_token on success, then routes to add-information.

4 · Verify

  • flutter pub get resolves with no version conflicts.
  • flutter test test/pkg/authentication is green.
  • Launch the app: splash → intro → login; a wrong password shows a 422 error toast; a correct one lands on home.
العربية

١ · أضف الاعتمادية

أضف اعتمادية مسار (path dependency) في ملف pubspec.yaml الخاص بالتطبيق المضيف (الحزمة موجودة تحت packages/authentication):

dependencies:
  authentication:
    path: packages/authentication

ثم نزّل الحزم:

flutter pub get

الحزمة نفسها بتعتمد على utd_app (عقد الميزات) وutd_studio_sdk (مُحرّك الشاشات المُدارة من السيرفر)، الاثنين كاعتمادية مسار.

٢ · سجّل الميزة

أضف AuthFeature() لقائمة الميزات اللي بتبنيها المنصّة. المصادقة ميزة أساسية (isCore = true) فسجّلها أولاً:

import 'package:authentication/authentication.dart';

List<AppFeature> buildFeatures() {
  return [
    AuthFeature(),   // أساسية — تُسجّل أولاً
    // ...باقي الميزات
  ];
}

التسجيل ده لوحده كفاية. الـ FeatureRegistry بيقوم بالباقي أوتوماتيك:

الـ Hookاللي بيوصّله
initialize()بيبني خدمة الـ API ← مصدر البيانات ← الـ repository ← الـ use cases، ويملأ AuthLocator (لأفعال Studio)، وينشئ الـ BLoCs. ويُنتظَر قبل بناء أي route.
getRoutes()بيرجّع AuthRoutes.routes() — splash / intro / login / register / add-information / recover-password / on-boarding / privacy.
getProviders()بيوفّر SplashBloc وLoginBloc وRegisterBloc وAddInformationBloc وRecoverPasswordBloc للتطبيق كله.
getTranslations()بيدمج كتالوج auth.* (إنجليزي + عربي) في ترجمات التطبيق (مع دمج اسم التطبيق).
للشاشات المُدارة من السيرفر، مرّر مُعالجات أفعال المصادقة (core.login / core.register / core.forgotPassword / core.completeProfile) إلى UtdStudio.init عبر StudioConfig.extraActions. بتقرأ الـ use cases من AuthLocator اللي بيملأه initialize(). ولو الشاشة مش منشورة، بتشتغل الصفحة الأصلية البديلة.

٣ · عقد الـ API في القاعدة

الحزمة بتتكلم مع القاعدة تحت بادئة /api. المسارات (من AuthApiService):

الـ use caseالـ endpoint
تسجيل الدخولPOST /auth/login
إنشاء حسابPOST /auth/register
إكمال المعلوماتPOST /profile/update
استعادة — إرسال OTPPOST /auth/forgot-password/send-otp
استعادة — التحقق من الرمزPOST /auth/forgot-password/verify-code
استعادة — إعادة التعيينPOST /auth/forgot-password/reset-otp

طلب وردّ تسجيل الدخول

تسجيل الدخول بيرسل { "type": "email_pass", "email", "password" }. عند النجاح لازم يحمل الردّ التوكن تحت data.auth_token (الحزمة بتقرأ json['auth_token'] داخل الجلسة وبتحفظه):

POST /api/auth/login
{ "type": "email_pass", "email": "user@x.com", "password": "secret" }

200 OK
{
  "message": "تم تسجيل الدخول",
  "data": { "id": 12, "is_first": false, "auth_token": "..." }
}

ردّ تسجيل الدخول مابيحملش كائن المستخدم الكامل (بس id / is_first / auth_token)؛ والحزمة بعدها بتحمّل المستخدم الحالي من /my-data داخل الجلسة.

بيانات اعتماد خاطئة ← 422

عند بيانات اعتماد غير صحيحة (أو فشل التحقق) لازم القاعدة ترجّع HTTP 422 مع message. الحزمة بتعرض الـ message ده كرسالة خطأ (toast) — وماتعيدش المحاولة. الـ register بالمثل بيرجّع data.auth_token الجديد عند النجاح، وبعدها يوجّه لإكمال المعلومات.
