UTD App · Flutter package

Authentication

Email login / register / password recovery / onboarding — clean architecture, BLoC, and server-driven (UTD Studio) screens with native fallback.

English

Overview

Authentication is the core auth feature for the UTD add-on platform. It owns the whole pre-login journey and talks to the base host's /api/auth/* endpoints. The Flutter package is authentication and the entry class is AuthFeature (registered with the platform's FeatureRegistry).

It is built as clean architecture: data (api service / datasource / repository impl) → domain (entities / params / use cases / repository contract) → presentation (BLoC + pages). Each screen is exposed as a route and renders the UTD Studio server-driven version when published, with the native page as a guaranteed fallback.

What it does

Login
Email + password → POST /auth/login, saves the token, hydrates the session, lands on home.
Register
Email + password → POST /auth/register, then routes to add-information.
Recover password
3-step email OTP flow: send code → verify code → set new password, with a resend cooldown timer.
Add information
Name / gender / birthday / avatar → POST /profile/update to complete the profile.
Onboarding
Swipeable intro slides ending in "Get Started".
Intro & Splash
Splash decides session vs intro; intro offers sign-in / create-account.

Routes & server-driven screens

AuthRoutes exposes the auth navigation paths. The intro, login, register and add_information routes wrap a StacDynamicScreen (from utd_studio_sdk): they render the published UTD Studio screen by name and fall back to the native page when nothing is published yet or the device is offline on first run — so auth never breaks before screens are synced+published.

The Studio glue (core.login / core.register / core.forgotPassword / core.completeProfile action parsers) reuses the same use cases via AuthLocator, so a tap inside a server-driven screen runs the identical success/failure flow as the native BLoC.

Architecture

State management

Each feature has its own BLoC: SplashBloc, LoginBloc, RegisterBloc, AddInformationBloc, RecoverPasswordBloc. They are created once in AuthFeature.initialize() and provided app-wide via getProviders().

Use cases & data

Use cases (LoginUseCase, RegisterUseCase, AddInfoUseCase, OTP-recovery use cases, ForgetPasswordUseCase) delegate to AuthRepository → AuthRemoteDataSource → AuthApiService, which holds the endpoint paths.

authentication/
├── lib/authentication.dart            # public barrel
├── lib/core/
│   ├── auth_impl.dart                  # AuthFeature (init / providers / routes / translations)
│   ├── auth_routes.dart                # AuthRoutes + StacDynamicScreen wiring
│   ├── auth_locator.dart               # use cases exposed to Studio actions
│   └── auth_strings.dart               # AR + EN translation catalog
└── lib/src/
    ├── data/        (api service · datasource · repository impl · models)
    ├── domain/      (entities · params · use cases · repository contract)
    └── presentation/(splash · intro · login · register · recover_password · add_information · on_boarding)

Translations

All strings are keyed under auth.* and shipped bilingual (EN + AR) by AuthStrings.translations(appName), surfaced to the platform via AuthFeature.getTranslations(). No hardcoded UI text — onboarding titles interpolate the app name.

العربية

نظرة عامة

Authentication هي حزمة المصادقة الأساسية في منصّة UTD. بتمتلك رحلة ما قبل تسجيل الدخول بالكامل وبتتكلم مع مسارات القاعدة /api/auth/*. اسم الحزمة في Flutter هو authentication والكلاس الرئيسي AuthFeature (بيتسجّل في FeatureRegistry الخاص بالمنصّة).

مبنية بأسلوب Clean Architecture: طبقة data (خدمة الـ API / مصدر البيانات / تنفيذ الـ repository) ← طبقة domain (الكيانات / المعاملات / الـ use cases / عقد الـ repository) ← طبقة presentation (الـ BLoC + الصفحات). كل شاشة معروضة كـ route وبترندر نسخة UTD Studio المنشورة لما تكون متاحة، مع الصفحة الأصلية (native) كبديل مضمون.

المسارات والشاشات المُدارة من السيرفر

AuthRoutes بيوفّر مسارات التنقّل. مسارات intro وlogin وregister وadd_information بتغلّف StacDynamicScreen (من utd_studio_sdk): بترندر شاشة UTD Studio المنشورة بالاسم، وبترجع للصفحة الأصلية لو مفيش شاشة منشورة أو الجهاز offline في أول تشغيل — عشان المصادقة ما تتعطّلش قبل مزامنة ونشر الشاشات.

الـ Studio glue (مُعالجات الأفعال core.login / core.register / core.forgotPassword / core.completeProfile) بتعيد استخدام نفس الـ use cases عبر AuthLocator، فالضغط جوّه شاشة سيرفر بيشغّل نفس تدفّق النجاح/الفشل بتاع الـ BLoC الأصلي.

المعمارية

إدارة الحالة

كل ميزة ليها BLoC خاص: SplashBloc وLoginBloc وRegisterBloc وAddInformationBloc وRecoverPasswordBloc. بتتعمل مرة واحدة في AuthFeature.initialize() وبتتوفّر للتطبيق كله عبر getProviders().

الـ use cases والبيانات

الـ use cases (LoginUseCase وRegisterUseCase وAddInfoUseCase و use cases استعادة الـ OTP وForgetPasswordUseCase) بتفوّض لـ AuthRepository ← AuthRemoteDataSource ← AuthApiService اللي بيحمل مسارات الـ endpoints.

الترجمات

كل النصوص مفهرسة تحت auth.* وبتشحن ثنائية اللغة (إنجليزي + عربي) عبر AuthStrings.translations(appName)، وبتتعرض للمنصّة من خلال AuthFeature.getTranslations(). مفيش نصوص واجهة مكتوبة بشكل ثابت — وعناوين الـ onboarding بتدمج اسم التطبيق.
